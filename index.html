Pre move

  1. Select an SSL certificate

    There are 3 main types of certificates you can choose from for your site:

    • Domain Validation (DV)
      • Standard SSL (Just as secure as other options).
      • Free via Let's Encrypt & Cloudflare, or purchase available.
      • Normally issued in minutes.
    • Oranization Validation (OV)
      • Shows organization within certificate but largly appears the same as DV.
      • Requires purchase.
      • Can take several days to issue.
    • Extended Validation (EV)
      • Shows oranization in url bar as well as within the certificate.
      • Requires purchase, more expensive.
      • Can take up to a week to issue.

    For most sites, DV certificates are exactly what you're looking for. But it's worth considering EV certificates as they show the organization in the url bar.

    Another consideration is do you need this certificate for a single domain/subdomain or for multiple. There are single domain, multi domain and wildcard (all subdomains) certificates available from most SSL providers, though Let's Encrypt & Cloudflare require you setup each subdomain seperatly.

    If you're just looking to secure your main domain a 1 or 2 DV certificates will work perfectly.
    Free DV certificates are available with Cloudflare
    Many sites such as SSLs.com offer certifciates.

    If you're interested in an OV or EV certificate, best to go ahead and purchase these in advance as they can take several days to be issued. They will also require proof of the organization's existence. But if you're selecting a DV certificate then you can wait till the moving day.

    Once you've selected the certificate you're interesed in you can move on to start migrating, but consider the following points to make sure you don't miss anything.

  2. Could you migrate first on a test site?

    If you are able to, first setting up a test / staging version of your site to migrate would ease the process greately as it reduces risk while making changes to the production site. This is optional as most sites don't have access to a test site, nor will many sites actually require one anyway.

  3. Dedicate some time to your migration

    Installing a certificate can be pretty quick, though the actual full migration process can take a good chunk of time to properly complete. We'd advise starting the moving day portion of this guide when you have a few hours free. If everythign goes smoothly it may take much less time, but it's worth having that spare time just in case.

  4. How will you track changes

    In the event that the migration causes some short term distruptions in traffic and/or search rank, you'll want stats to fall back on and provide visibility of what exactly changed.

    If you havent already, setup Google Analytics or a similar service and let it run for at least a week to get a good idea of current traffic & behaviour.

    It's also worth making sure you have Google Search Console (Webmaster tools) setup to track indexed pages and keywords used to find your site along with average position. You may also want to check out Bing's Webmaster Tools.

  5. Consider making some changes pre migration

    Some changes such as fixing mixed content can often be done before the move.
    We've listed it later on as part of the migration flow, but feel free to skip ahead through the guide to ease the changes later.
    Other changes such as redirects & 3rd party changes must wait 'til you first have SSL installed.

  6. Do you plan to move your entire site or just specific areas first?

    If you can, moving your entire site. It means more work up front but given the SEO benifits and knowelege that your entire site is encrypted for users is worth it.

    However, not all sites are able to do this at one, at the very least you'll need to setup HTTPS for any login forms, payment pages and any pages that require or reveal user details.

  7. Does you site have multiple variations?

    During the migration you'll need to be aware of all variations of your site such as multiple lanages or location based changes or even AB tests in progress? During the mixed content fixing step you'll need to check all variations.

  8. Using a CDN?

    Check out your CDN's site to see if there's anything you need to change / be aware of during the migration process.

  2. Check the certificate

    First check, load the site up in your browser and change the url to https. https://www.ssllabs.com/ssltest/

  3. Fix mixed content

    - CSP upgrade-insecure-requests (http://caniuse.com/#feat=upgradeinsecurerequests) (browser support) - CSP report issues (https://report-uri.io or https://httpschecker.net/guides/https-reporter - wordpress plugin (https://wordpress.org/plugins/really-simple-ssl/ or https://wordpress.org/plugins/ssl-insecure-content-fixer/) - desktop: https://httpschecker.net/guides/https-checker - cli: https://github.com/bramus/mixed-content-scan - a manual check is always a good start and chrome dev tools will show issues on each page

  4. Setup HTTP to HTTPS redirect

    Setup your .htaccess file to redirect to HTTPS

  5. Change platform settings

    In Settings > General > Update the "WordPress Address (URL)" and "Site Address (URL)" to be https:// instead of http://

    In System > Global Configuration > Server > Force SSL > Select Entire Site

    In Admin > System > Configuration > General > Web > Secure > Select Yes for "Use Secure URLs in Frontend" & "Use Secure URLs in Admin"

    In Admin > Online Store > Domains > SSL certificates > click "Activate SSL certificates"

    Check if your platform has a setting area for the URL or using SSL.

  6. Update 3rd party analytics & other links

    Some 3rd party tracking sites require you either update the tracked origin url or setup a new account for monitoring.

    Here's a few examples, but worth doing a quick audit of any 3rd party tools you use:

    • Google Search Console (Webmaster Tools) - Requires you setup a new account, this allows you to monitor traffic still in Google's index hitting the HTTP version first vs the HTTPS version of your site
    • Google Analytics - Requires a URL change in a couple areas:
      • Admin > View Settings > Website's URL
      • Admin > Property Settings > Default URL
      • Admin > Property Settings > Search Console > Adjust Search Console > Link to the new Search Console account
    • Google Merchant Center (Shopping) - Requires claiming the new URL in Business information > Website > Claim your website

Post move

  1. Wait a day or even a week...

    If you're using CSP this is a great point to check reports and fix any issues, this can be an ongoing process.

  2. Impliment HSTS (HTTP Strict-Transport-Security) to make sure users always see the SSL version

    One of the main reasons to wait a day or so is to make sure you don't need to back out of this.
    With HSTS you can tell browsers when users visit your site to always go staright to the HTTPS version.

    Setup ...

  3. Plan for when the certificate expires

    You should make a note of when your certificate expires on a calander to be safe. Even if you've setup automatic renewal of the certificate with lets encrypt, you may still want to be alerted to check it's worked. With other SSL providers, they normally email you before renewal but best to be safe and keep track of this date yourself. If you had to install the certificate yourself, consider making notes on how you did this for your future self to use.

Going above and beyond

  1. Update internal links in pages, emails etc.

    Even with a redirect from HTTP to HTTPS setup, it's best to minimise temp HTTP redirects as these can still lead to exposing what the user was doing.

  2. Update links on social media etc.

    Though you've setup all your redirects, it's still best to try to send users direct to the HTTPS version without need to go through redirects.
